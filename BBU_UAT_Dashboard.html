Defect Analysis Tab — Quick Reference

BBU o9 TPO/TPM Implementation · UAT Round 1 Known Defects Dashboard

For internal use only
Dashboard v35 · April 2026
The Defect Analysis tab gives you a structured view of all UAT defects categorized by type — not just by test script. Use it to understand the shape of the defect backlog, identify the highest-risk functional areas, and prioritize resolution efforts across the team.

How to Access

1
Open the BBU UAT Known Defects dashboard in your browser.
2
Click the 📊 Defect Analysis tab in the black navigation bar — third tab, after Known Defects and Test Completion.
3
The tab defaults to Active defects only. Toggle to All (top right) to include closed defects.

Navigating the Tab

1
Click any category card to filter the view. A bar chart appears showing defect counts by test script.
2
Click a bar to drill into a specific test script. All other bars collapse and the table updates.
3
Navigate back via the breadcrumb — click the category name to go up one level.
4
Click any ADO # to open that defect directly in Azure DevOps.
⚠️ Potential Duplicates: A yellow banner means some defects in the current view share the same test step as another active defect — they may be the same root issue filed twice.

The Five Categories

🔧 FunctionalSystem does not behave as specified — broken UI, missing buttons, bad defaults, download failures.
⚙️ Business RulesCode works but business logic is wrong — override precedence, misconfigured rules, misalignment.
🔄 Process / E2EWorks step-by-step but fails across a full workflow — carry-forward failures, broken handoffs.
🗄️ Data IntegrityData behaves incorrectly over time — past weeks editable, LY/LLY wrong, historical locks missing.
🔗 IntegrationUpstream/downstream failures — ERP cost alignment, sync delays, master data inconsistency.

Tips

🎯
Start with Functional. This category typically contains the highest defect volume and is the best place to get an overall read on system quality.
📍
Use the bar chart to find hotspots. After selecting a category, the bars show which test scripts are driving the most defects — longest bars need the most attention.
🔄
Data refreshes automatically when an admin uploads a new Smartsheet export via the ⚙ Admin link. All counts and charts update instantly.

Known Defects Tab — Quick Reference

BBU o9 TPO/TPM Implementation · UAT Round 1 Known Defects Dashboard

For internal use only
Dashboard v35 · April 2026
The Known Defects tab is the primary reference for all UAT defects logged during Round 1. It shows every defect across all test scripts with full detail — description, expected behavior, status, and severity. Use filters and search to find exactly what you need without scrolling through the full list.

Filters — Status & Severity

▶ Active Default view. Shows all open defects — excludes Closed only.
🔴 High Active defects marked High severity only.
🟡 Medium Active defects marked Medium severity only.
🟢 Low Active defects marked Low severity only.
⏳ Pending Defects submitted for fix validation — progress at 75% or 90% in ADO.
⚠️ Dupes Active defects that share the same test step as another active defect. Review before re-reporting.
✅ Closed Resolved defects. Rows appear grayed out.
All Every defect regardless of status or severity.

Phase Filter

Use the E2E + UAT / E2E Only / UAT Only toggle to separate defects found during end-to-end testing (by the project team) from those found during UAT (by end users). Combining with a severity filter gives the most targeted view.

Status Indicators

Active Open defect, under investigation or pending fix.
New Recently filed, not yet picked up in ADO.
Pending Fix submitted, awaiting tester validation.
Closed Validated as resolved. Row is grayed out.

Search

🔍
Search defects — searches across all fields (title, description, ADO number, tester name, step ID). Works across all filter states simultaneously.
👤
Find my pending items — type your name to see only Pending Validation defects assigned to you. Use this to check what you need to re-test.
The two search boxes are mutually exclusive — using one clears the other.

Navigation & Layout

1
Jump to section — use the dropdown below the tab bar to jump directly to a test script section. The section expands automatically on jump.
2
Expand / Collapse — sections are collapsed by default. Click any section header to expand it, or use the Collapse All / Expand All buttons.
3
ADO link — click any ADO number in the table to open that defect directly in Azure DevOps in a new tab.
4
Section counts update dynamically as you change filters — the number shown reflects what's currently visible, not the total.

Table Columns

Phase ADO # Step ID Status Title Defect Description Expected Behavior Severity
Tester name appears below the title with a 👤 icon. Potential duplicate defects show a ⚠️ icon — hover it to see which other ADO numbers share the same step.
⚠️ Before filing a new defect: check the Potential Duplicates filter and search by step ID to confirm the issue hasn't already been logged by another tester.

Tips

🎯
Active is your default. The view opens on Active defects — everything open, nothing closed. Switch to a severity filter when you want a more targeted list.
👤
Check your pending items regularly. Use the "Find my pending items" search with your name to see what's waiting for your validation before the sprint closes.
🔄
Data refreshes automatically when an admin uploads a new Smartsheet export via the ⚙ Admin link. All counts and statuses reflect the latest file.

Test Completion Tab — Quick Reference

BBU o9 TPO/TPM Implementation · UAT Round 1 Known Defects Dashboard

For internal use only
Dashboard v35 · April 2026
The Test Completion tab tracks progress across all UAT Round 1 test scripts and testers. It shows how many assigned tester-script combinations have been completed, how the team is pacing against the sprint target, and where completion gaps exist — both by test script and by individual tester.

The Four Gauges

Each gauge shows a percentage with a burn bar underneath. The burn bar compares actual progress to where the team should be based on elapsed sprint days.
Overall Complete % — total tester-script combinations marked complete, divided by total assigned. This is the headline number.
Tests Started % — how many of the 27 test scripts have at least one tester who has completed them. Low here means whole scripts haven't been touched yet.
Testers Active % — how many assigned testers have completed at least one script. Low here means some testers haven't started at all.
Avg Tester Complete % — the mean completion percentage across all testers. Useful for spotting whether progress is concentrated in a few people or spread across the group.
Reading the burn bar: The dark tick mark shows where completion should be based on elapsed sprint days. A green label means ahead of pace, red means behind. The gap number shows the difference in percentage points.

Completion Logic

A tester-script pair counts as complete when the Smartsheet export shows a status of Completed or Completed with Defects for that combination. Scripts completed with defects still count — the defects are tracked separately in the Known Defects tab. Each tester-script pair is counted once regardless of how many defects were filed.

By Test Sub-tab

Shows all 27 test scripts as rows with a progress bar and completion count. Colors indicate status:
0% — not started
Below 50% — in progress, low coverage
50–84% — in progress, good coverage
85%+ — near or at full completion
Click the completed count on any row to expand it and see which testers are done (green chips) and which still have that script outstanding (red chips).

By Tester Sub-tab

A full heatmap — every tester as a row, every test script as a column. Each cell shows one of three states:
✓
Completed
○
Assigned, not done
—
Not assigned
Each tester row shows their done/assigned count and completion percentage, color-coded by progress level. Use the search box to filter by name, and the sort buttons to reorder by name or completion percentage.
Note: The heatmap uses UAT-phase data only. E2E testing rows do not appear here because completion status is not tracked at the tester level for E2E scripts.

Tips

📊
Watch the burn bars. The gauges tell you where you are; the burn bars tell you whether you're on track. A red gap on Overall Complete % is the first signal that the sprint is at risk.
🔍
Use By Tester to find stragglers. Sort by % Low→High to surface testers with the least progress. Cross-reference with By Test to see if their outstanding scripts are high-priority ones.
🔄
Data refreshes automatically when an admin uploads a new Smartsheet export via the ⚙ Admin link. All gauges, bars, and the heatmap update to reflect the latest file.
